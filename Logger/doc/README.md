# Logger 工程总结

## 工程中的问题与挑战


## 收获
- 对于有限且固定的状态，使用 enum class 完美符合语义；
- 设计的时候要注意，提供给外部的接口和内部实现接口，设计思路是不一样的，不应该混为一谈；
- 真实开发流程：需求 -> 场景 -> 职责 -> 接口 -> 数据 -> 流程 -> 实现 -> 重构；
- 对于每一个模块的设计固定使用一套自洽的思考流程：
  - 组件解决什么问题？组件职责。
  - 谁使用这个组件？使用场景。
  - 用户如何使用？对外接口。
  - 为了保证功能，应该持有哪些数据/状态？内部状态/成员变量。
  - 数据如何流动，完整的运行流程？核心执行流程。
  - 类的初步设计。
- You Are‘t Gonna Need It!不要为了还不存在的需求而增加今天的复杂度，设计和实现应该先遵循当前的需求，不要过度设计；
- 接口设计的时候应该注意职责的简化，一个接口的参数如果越来越多，说明接口的职责已经混乱了。需要注意将行为和配置分开，行为的接口应该只接收该行为所必需的参数即可，而配置相关的应该交由其他接口设置；
- 对于成员变量的思考，主要在三个方面：配置、状态、资源。其中着重考虑每一个组件真正持有的资源是什么，这是最核心的；
- 后续的学习遵循三个阶段：
  - 盲设计：我简单了解组件的需求、场景，不查询资料直接自己开始设计，这个过程会暴露我设计上的一些缺陷以及我对于组件的认识；
  - GPT 审查：gpt review 我当前的设计，并且讨论不足、缺陷，尝试修改弥补；
  - 成熟设计学习：完成我自己的成熟版本后，阅读现有的成熟设计，理解其中很多内容是为了解决我当时的什么设计问题。
- 区分配置和状态，配置是在构造的时候就确定的内容，一般不会在程序运行过程中被修改的，而状态是可以修改的，用于描述运行过程中的状态；
- 对于是否应该避免拷贝，应该先看语义，后看成本，主要考虑以下内容：
  - 是资源还是值？对于资源型对象不能够拷贝，因为这个在语义上是错误的；对于值对象，则是符合拷贝的语义的；
  - 拷贝是否昂贵？对于一些 Image、Database 等真正大的对象，需要考虑成本；
  - 该对象是否天然应该拥有自己的副本？例如，对于 Logger 而言，它天然应该持有自己的配置（副本），而不应该让自己的配置可以因为外部对象而改变。
- Pass by Value + Move: 传值的时候，注意不要多余拷贝，该直接移走的资源直接使用 move 拿走；
