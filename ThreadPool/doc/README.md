# ThreadPool 工程总结
## 需求背景
ThreadPool 是一个线程管理与启动工具，上层初始化线程池后，只需要将任务 submit 至线程池即可，不需要关注其余细节，然后在结束使用后，只需要 stop 线程池。

## 模块职责划分

ThreadPool：
- 管理工作线程生命周期
- 管理线程池运行状态
- 对外提供 submit / stop 接口

BlockQueue：
- 管理任务生命周期
- 提供线程安全的 push / pop
- 管理等待与唤醒
- 管理 close 状态

## 生命周期与资源管理
ThreadPool 包含自己在内，有三个核心资源：自己本身（ThreadPool）、工作线程（m_threads）、任务队列（m_taskQueue）。其中任务由上层 submit 交给 ThreadPool，然后 ThreadPool 再将任务交给 m_taskQueue，由任务队列管理，交给空闲工作线程管理。

## 核心接口具体实现
### submit
首先明确上层 submit 的任务本质是一个个函数，然后不同的任务函数会有不同的参数，所以这是一个模板函数，而且是一个变参数模板。然后因为 submit 函数本身不消费这些内容，只是将这些内容包装并交给任务队列，后续由任务队列管理，所以这里需要完美转发，函数签名也应该使用万能引用。

接着就是具体对于 task 的包装，为了所有的 task 函数都能够进入任务队列，我们需要将 task 包装为 function<void()> 类进行类型擦除，而任务队列中的任务明面上也是各种 function<void()> 对象。

然后谨记 submit 是个包装站、中转站，所以实现过程中要注意完美转发，不要损失内容。包装好后直接交给任务队列即可。

### stop
这个函数承担三个任务，注意实现的幂等性：
- 更新线程池状态位；
- 关闭任务队列、更新任务队列状态；
- 汇合、关闭所有线程。

### 构造函数
构造函数这里要注意委托构造，当没有参数的时候默认线程数为 1，直接调用线程数为 1 的构造函数即可。

- 完美转发 、万能引用。
