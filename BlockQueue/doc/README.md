# BlockQueue 工程总结
## 需求背景
在多线程环境下，使用 mutex 和 queue 的配合方式，会出现大量的 CPU 性能浪费，没有任务时，queue 仍需始终轮询忙等待。  
而 BlockQueue 则是一个支持多线程并发使用的队列，在没有任务时，能够自动休眠线程，当有任务到达时能够自动唤醒线程领取任务。

## 实现思路
BlockQueue 的关键实现在于能够在合适的时机睡眠或者唤醒线程，这主要依赖于 std::condition_variable 来实现。  
std::condition_variable 类通过和 unique_guard 的配合，提供了 wait、notify_one、notify_all 等方法。其中 wait 方法会根据条件，若不满足先解锁然后睡眠线程，等待被唤醒，苏醒后则继续上述判断，如果满足条件且抢到锁则结束 wait 往下运行，注意整个条件判断都是在有锁的背景下进行的；notify_one 方法则是唤醒一条正在睡眠的线程；notify_all 则是唤醒所有线程。  

## 核心接口具体实现
### push
push 方法由生产者线程调用，将任务推入队列。要点：
- push 方法一定不能在队列 close 后调用，所以需要做判断。这里做了简单处理，通过 bool 值简单反映 push 是否正确合法，交由上层来处理，这也是指责边界的体现所在。底层的 BlockQueue 之管理自身功能状态。
- push 后固定 notify_one。这里是存在一定的资源浪费的，因为生产者 push 任务后，不一定需要唤醒一条线程，可能会正好被其他线程领取，但是这是基于实现复杂度及正确性的考虑。或者考虑价格判断，当队列之前为空，正好 push 第一个任务时才 notify_one。但其实这个过程中的线程被多余唤醒的可能性及相关性能浪费本身也比较低，可以忽略不记（但如果线程数非常大呢？）。
- 注意 notify 不应该在锁内，不然容易造成线程被唤醒，等待锁，等待时间较长然后又睡眠的情况，线程虚假苏醒。

### pop
pop 方法由消费者线程调用，领取任务。要点：
- 通过 conditon_variable 的 wait 方法使线程在队列为空且队列未 close 时睡眠，而不用反复轮询，浪费 CPU 性能。
- 被唤醒并通过 wait 后，实际队列状态可能有以下三种：队列不为空且未 close、队列不为空且 close、队列为空且 close。队列不为空时，都应该先继续领取任务，只有队列为空且 close 后才意味着线程任务结束，此处通过返回空值表达该含义。
- 通过以上逻辑注意到实际上返回值可能为空，也可能是具体的任务，这里采用了 C++17 的 std::optional 来处理这种情况。
- 通过这个方法的实现过程，重点体会状态判断、操作原子性判断的思考路径。

### close
close 方法由主线程调用，控制队列状态，表示再无新任务入队列。要点：
- 调整状态位。
- close 后应该 notify_all，唤醒所有线程，避免队列关闭后，仍有线程在睡眠造成主线程无法及时 join 所有线程。而这本身也是 close 状态的核心作用，避免线程无限期的睡眠。

### 其他补充
- 构造函数、析构函数采用默认即可。但是析构函数作为最后一道防线，应该通过 assert 避免队列在一些非法情况下被提前析构，如队列还未 close、队列不为空时，应该提醒开发者有疏漏。
- BlockQueue 应该禁止拷贝，因为 BlockQueue 本身持有锁，而锁是绑定在操作系统底层的物理资源，具有排他性，本身是不能拷贝的；而且 BlockQueue 本身是一个实体对象，代表的是系统中一个独一无二的运作机制或是物理存在，其语义也是禁止拷贝的。这里补充禁止拷贝的考虑方式：
  - 管理了独占的系统资源，如互斥锁、线程、网络套接字等；
  - 语义上是独一无二的实体对象，而不是值对象；
  - 资源本身较大，不可轻易分割时也应该禁止拷贝。

## 收获
- 多线程、并发程序，应该多思考边界情况，并且关注哪些操作是原子操作，然后在每一个原子操作之间都应该考虑状态的波动变化。
- 测试一般要有基准测试、边界测试...
